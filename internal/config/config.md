# Scratchpad directory ŌĆö configuration guide

This directory is a self-contained Scratchpad store. Everything the tool needs lives
here: this guide, the marker config, the pads, and the runtime socket.

> This guide is generated by the tool (embedded in the binary and written on
> init/bootstrap). It is documentation only ŌĆö to change settings, edit
> `scratchpad.config.json`, not this file.
>
> Examples use the canonical command name `scratchpad`; if you renamed the
> binary, substitute your name.

## Layout

```
<this dir>/
Ōö£ŌöĆŌöĆ scratchpad.config.json    # marker + settings (this dir is "initialized" iff it exists)
Ōö£ŌöĆŌöĆ config.md                 # this guide
Ōö£ŌöĆŌöĆ scratchpad.sock           # unix socket while `serve` runs (derived: <instance>.sock)
ŌööŌöĆŌöĆ projects/
    ŌööŌöĆŌöĆ <project>/<padid>.md  # one markdown file per pad
```

The socket and `projects/` paths are **derived** from this directory ŌĆö they are never
configured separately. Move the whole directory and everything moves with it.

Pads are plain markdown: you can `cat`, `grep`, and `rm` them directly. Scratchpad
keeps no state outside the pad files ŌĆö deleting a file deletes the pad, cleanly.

## How this directory is found

Every command resolves the Scratchpad dir with this precedence (high ŌåÆ low):

1. `--dir <path>` flag
2. `SCRATCHPAD_DIR` environment variable
3. the `dir` field in the marker file at the DEFAULT location (`~/.scratchpad`)
4. the default `~/.scratchpad` itself

The default dir is **auto-bootstrapped** on first use (created with this guide, the
marker, and an empty `projects/`). An *explicit* dir (1ŌĆō3) must already have been
created with `scratchpad init --dir <path>` ŌĆö commands refuse to auto-create it, so
a typo can never silently seed a store in the wrong place.

## The marker file (`scratchpad.config.json`)

```json
{
  "type": "scratchpad",
  "version": 1,

  "display_name": "Scratchpad",
  "instance": "scratchpad",

  "dir": "",
  "default_project": "default",

  "limits": {
    "max_title_kb": 4,
    "max_content_kb": 64,
    "max_sections_per_pad": 1000,
    "max_pads_per_project": 1000
  },
  "wait": { "default_s": 60, "max_s": 300 },

  "tcp": {
    "port": 6710,
    "token_digests": ["sha256:..."],
    "allowed_origins": []
  }
}
```

Field reference:

- **`type`** (required) ŌĆö always `"scratchpad"`. Identifies the file format; a
  different value is rejected.
- **`version`** (required) ŌĆö marker schema version. A file newer than the binary
  understands is refused with an upgrade hint.
- **`display_name`** ŌĆö human-facing name of this deployment. Default `"Scratchpad"`.
- **`instance`** ŌĆö technical label; names the unix socket (`<instance>.sock`).
  Default `"scratchpad"`.
- **`dir`** ŌĆö optional pointer that relocates the store to another **initialized**
  directory. Only meaningful in the marker at the default location
  (`~/.scratchpad`); use it to re-home the whole machine's store (for example onto
  an encrypted volume) without setting an env var everywhere.
- **`default_project`** ŌĆö project used when a command/tool doesn't pass one. Default
  `"default"`. Overridden per run by `SCRATCHPAD_PROJECT_NAME` or `--project`.
- **`limits`** ŌĆö optional resource bounds; omitted fields use the defaults shown above.
  `max_title_kb`/`max_content_kb` bound one section's title/content size;
  `max_sections_per_pad`/`max_pads_per_project` bound growth.
- **`wait`** ŌĆö optional MCP `pad_wait` timing: `default_s` when the caller omits
  `timeout_s`, `max_s` the server-side cap (values above it are clamped). The CLI
  `pad wait` is not affected by this cap.
- **`tcp`** ŌĆö optional settings for the opt-in `serve --tcp` transport: the loopback
  `port` (default 6710), `token_digests` (bearer tokens stored as `sha256:<hex>`
  digests ŌĆö never the raw token), and `allowed_origins` (exact-match Origin
  allow-list for browser-based clients; empty rejects cross-origin browsers).
  Having a `tcp` group here does NOT start TCP ŌĆö only `serve --tcp` does; the group
  just supplies its settings.

Only `type`, `version`, `display_name`, and `instance` are written at init; add the
optional groups when you need them.

**Not stored here on purpose:** paths (`projects/`, socket ŌĆö derived from the dir),
the author identity (per-agent; set `SCRATCHPAD_AUTHOR` in each agent's
environment), and pad passwords (each protected pad carries its own bcrypt hash in
the pad file's header ŌĆö removing the file removes every trace of it).

## Environment variables

| Variable | Meaning | Default |
|---|---|---|
| `SCRATCHPAD_DIR` | Scratchpad dir (see resolution above) | `~/.scratchpad` |
| `SCRATCHPAD_PROJECT_NAME` | default project when none is passed | `default` |
| `SCRATCHPAD_AUTHOR` | default author for the CLI `--as` flag | ŌĆö |
| `SCRATCHPAD_NONINTERACTIVE` | truthy = never prompt (automation) | ŌĆö |

Every variable has a matching flag; on conflict **flag > env > marker file > default**.

## Pad file format

One pad = one markdown file `projects/<project>/<padid>.md`:

```markdown
<!-- scratchpad v1; created: 2026-07-11T10:29:00Z; password: $2b$12$... -->

# 1 - frontend - How does API X work
<!-- ts: 2026-07-11T10:30:00Z -->

Question bodyŌĆ”

# 2 - backend - Answer about API X
<!-- ts: 2026-07-11T10:42:15Z -->

Answer bodyŌĆ”
```

- The first line is the pad header. `password:` (a bcrypt hash) appears only for
  protected pads ŌĆö access control only; the content stays plaintext.
- Each post is a section headed `# <n> - <author> - <title>`; only lines matching
  that exact pattern count as section boundaries.
- Turn state is derived from the last section (its author may not post next). There
  is no state anywhere else: deleting the last section by hand hands the turn back.
